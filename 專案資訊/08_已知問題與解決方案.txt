問題 1.1.1: 任務創建不與使用者關聯
──────────────────────────────────
發現日期: 2025-05-03
嚴重程度:  嚴重
影響範圍: 任務管理、資料隔離

問題描述:
  創建任務時未正確綁定 userId，導致任務無法正確歸屬於使用者。
  其他使用者可能看到不屬於自己的任務。

根本原因:
  - TodoViewModel 缺少使用者身份驗證監聽器
  - TodoTask 模型處理空 userId 的邏輯不正確
  - Firebase Auth 狀態變更時未及時更新資料

解決方案:
  1. 在 TodoViewModel 添加使用者身份驗證監聽器
     ```swift
     init() {
         // 監聽認證狀態變更
         Auth.auth().addStateDidChangeListener { [weak self] _, user in
             guard let self = self else { return }
             self.currentUserId = user?.uid
             if user != nil {
                 Task {
                     await self.fetchTasks()
                 }
             } else {
                 self.tasks = []
             }
         }
     }
     ```

  2. 修改 TodoTask 模型，確保強制要求 userId
     ```swift
     struct TodoTask: Identifiable, Codable {
         var id: String = UUID().uuidString
         var userId: String  // 移除可選性
         var title: String
         // ...

         init(title: String, userId: String) {
             self.id = UUID().uuidString
             self.userId = userId
             self.title = title
             // ...
         }
     }
     ```

  3. 在創建任務時強制驗證使用者登入
     ```swift
     func addTask(_ task: TodoTask) async {
         guard let currentUserId = Auth.auth().currentUser?.uid else {
             errorMessage = "請先登入"
             return
         }

         var newTask = task
         newTask.userId = currentUserId

         // 繼續創建任務...
     }
     ```

驗證方法:
   創建任務後檢查 Firestore 中的 userId 欄位
   切換使用者帳號，確認任務隔離
   未登入狀態無法創建任務

檔案位置:
  - studyAssistant/ViewModels/TodoViewModel.swift
  - studyAssistant/Models/TodoTask.swift


問題 1.1.2: 重開專案時無法從資料庫更新資料
───────────────────────────────────────
發現日期: 2025-05-03
嚴重程度:  嚴重
影響範圍: 資料持久化、多設備同步

問題描述:
  應用關閉後重新開啟，無法載入之前儲存的任務資料。
  Firestore 中有資料，但 App 顯示為空。

根本原因:
  - 資料庫結構設計不一致
  - 任務儲存路徑與讀取路徑不匹配
  - 資料路徑: tasks/{taskId} (舊結構)
  - 讀取路徑: tasks/{userId}/userTasks/{taskId} (新結構)

解決方案:
  1. 重新設計資料庫結構，統一路徑
     舊結構:
     tasks/
       └── {taskId}

     新結構:
     tasks/
       └── {userId}/
           └── userTasks/
               └── {taskId}
                   └── instances/
                       └── {instanceId}

  2. 添加數據遷移功能
     ```swift
     func migrateOldTasks() async {
         // 讀取舊資料
         let oldTasksSnapshot = try? await db.collection("tasks").getDocuments()

         guard let documents = oldTasksSnapshot?.documents else { return }

         for document in documents {
             guard let task = try? document.data(as: TodoTask.self),
                   !task.userId.isEmpty else {
                 continue
             }

             // 寫入新結構
             try? await db.collection("tasks")
                 .document(task.userId)
                 .collection("userTasks")
                 .document(task.id)
                 .setData(from: task)

             // 刪除舊資料 (可選)
             try? await document.reference.delete()
         }
     }
     ```

  3. 在應用啟動和用戶登入時執行遷移
     ```swift
     func application(_ application: UIApplication,
                      didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
         FirebaseApp.configure()

         // 執行資料遷移 (僅一次)
         if !UserDefaults.standard.bool(forKey: "didMigrateData") {
             Task {
                 await migrateOldTasks()
                 UserDefaults.standard.set(true, forKey: "didMigrateData")
             }
         }

         return true
     }
     ```

驗證方法:
   重新安裝 App，資料仍然存在
   多設備登入同一帳號，資料一致
   Firestore Console 中檢查資料路徑正確

預防措施:
  - 在專案初期確定資料庫結構並文檔化
  - 使用常數管理集合路徑
  - 定期備份資料庫

檔案位置:
  - studyAssistant/Services/FirebaseService.swift
  - studyAssistant/studyAssistantApp.swift


問題 1.1.3: DataServiceProtocol 協議一致性問題
─────────────────────────────────────────
發現日期: 2025-05-02
嚴重程度:  中等
影響範圍: 服務層架構

問題描述:
  FirebaseService 類實作不符合 DataServiceProtocol 協議的定義。
  編譯錯誤: Type 'FirebaseService' does not conform to protocol 'DataServiceProtocol'

根本原因:
  - getTimerRecords 和 getTimerStatistics 方法的參數不符合協議定義
  - 協議定義與實作不同步

解決方案:
  1. 修正協議定義 (DataServiceProtocol.swift)
     ```swift
     protocol DataServiceProtocol {
         func getTimerRecords(userId: String) async throws -> [TimerRecord]
         func getTimerStatistics(userId: String) async throws -> TimerStatistics
     }
     ```

  2. 確保 FirebaseService 實作符合協議
     ```swift
     class FirebaseService: DataServiceProtocol {
         func getTimerRecords(userId: String) async throws -> [TimerRecord] {
             // 實作
         }

         func getTimerStatistics(userId: String) async throws -> TimerStatistics {
             // 實作
         }
     }
     ```

驗證方法:
   專案成功編譯，無協議一致性錯誤
   所有方法實現與協議定義一致

預防措施:
  - 修改協議時，同步更新所有實作類別
  - 使用 Protocol Witness Table 檢查一致性
  - 定期執行編譯檢查

檔案位置:
  - studyAssistant/Services/DataServiceProtocol.swift
  - studyAssistant/Services/FirebaseService.swift

問題 1.2.1: UserProfile 初始化問題
─────────────────────────────────
發現日期: 2025-05-02
嚴重程度:  中等
影響範圍: 使用者資料管理

問題描述:
  使用 UserProfile 構造函數時傳入不正確的參數數量，導致初始化失敗。

根本原因:
  - 直接使用 UserProfile() 初始化，未提供必要參數
  - ID 和 email 未正確設置

解決方案:
  1. 建立靜態工廠方法
     ```swift
     extension UserProfile {
         static func defaultProfile() -> UserProfile {
             return UserProfile(
                 id: UUID().uuidString,
                 email: "",
                 username: "",
                 motivationalQuote: nil,
                 targetDate: nil,
                 userGoal: nil,
                 learningStage: nil,
                 isVIP: false,
                 lastLoginAt: Date(),
                 version: 1,
                 lastSyncedAt: nil,
                 isDeleted: false
             )
         }

         static func fromFirebaseUser(_ user: User) -> UserProfile {
             return UserProfile(
                 id: user.uid,
                 email: user.email ?? "",
                 username: user.displayName ?? "",
                 motivationalQuote: nil,
                 targetDate: nil,
                 userGoal: nil,
                 learningStage: nil,
                 isVIP: false,
                 lastLoginAt: Date(),
                 version: 1,
                 lastSyncedAt: nil,
                 isDeleted: false
             )
         }
     }
     ```

  2. 使用工廠方法替代直接初始化
     ```swift
     //  錯誤
     let profile = UserProfile()

     //  正確
     let profile = UserProfile.defaultProfile()
     // 或
     let profile = UserProfile.fromFirebaseUser(currentUser)
     ```

驗證方法:
   成功創建 UserProfile 實例
   ID 和 email 正確設置

檔案位置:
  - studyAssistant/Models/UserProfile.swift

發現日期: 2025-05-02
嚴重程度:  輕微
影響範圍: 應用設定管理

問題描述:
  fetchAppSettings 和 saveSettings 方法中引用了 AppSettings.userId，
  但該屬性不存在於 AppSettings 模型中。

根本原因:
  - AppSettings 模型設計時未包含 userId
  - 使用 Firestore 路徑 settings/{userId} 儲存，而非模型欄位

解決方案:
  1. 移除對 AppSettings.userId 的引用
  2. 使用當前登入用戶 ID

  ```swift
  func fetchAppSettings() async throws -> AppSettings {
      guard let userId = Auth.auth().currentUser?.uid else {
          throw DataServiceError.unauthenticated
      }

      let document = try await db.collection("settings")
          .document(userId)  // 使用當前用戶 ID
          .getDocument()

      if document.exists {
          return try document.data(as: AppSettings.self)
      } else {
          // 回傳預設設定
          return AppSettings(
              id: userId,
              isDarkMode: false,
              notificationsEnabled: true,
              isShockEnabled: false,
              lastModified: Date()
          )
      }
  }
  ```

驗證方法:
   成功載入和儲存應用設定
   設定與當前登入用戶正確關聯

檔案位置:
  - studyAssistant/Services/FirebaseService.swift

  進入 ProfileSettingView 時，無法顯示已儲存的個人資料。

根本原因:

     ```swift
     // UserSettingsViewModel.swift

         public func loadData() async {

             defer { isLoading = false }

                 userProfile = try await firebaseService.getUserProfile()

                 print("載入資料失敗: \(error)")
             }
         }
     }
     ```

     ```swift
     struct ProfileSettingView: View {
     ```

     ```swift
     if viewModel.isLoading {
     ```

問題 1.3.1: 每日任務會導致每天都變成已完成
─────────────────────────────────────────────
發現日期: 2025-05-06
嚴重程度:  嚴重
影響範圍: 重複任務功能

問題描述:
  完成一個每日重複任務後，所有日期的實例都標記為已完成。

根本原因:
  - 所有重複任務實例共用同一個 isCompleted 狀態
  - 缺少任務實例(TaskInstance)的獨立狀態管理

解決方案:
  1. 引入 TaskInstance 資料結構
     ```swift
     struct TaskInstance: Identifiable, Codable {
         var id: String = UUID().uuidString
         var parentTaskId: String
         var date: Date
         var isCompleted: Bool
         var createdAt: Date
     }
     ```

  2. 在 Firestore 中建立 instances 子集合
     ```
     tasks/{userId}/userTasks/{taskId}/
       └── instances/
           └── {instanceId}
               ├── parentTaskId: String
               ├── date: Timestamp
               ├── isCompleted: Boolean
               └── createdAt: Timestamp
     ```

  3. 修改任務完成邏輯
     ```swift
     func toggleTaskCompletion(taskId: String, instanceId: String?, isCompleted: Bool) async {
         if let instanceId = instanceId {
             // 更新特定實例
             try? await firebaseService.updateTaskInstance(
                 taskId: taskId,
                 instanceId: instanceId,
                 isCompleted: isCompleted
             )
         } else {
             // 更新非重複任務
             try? await firebaseService.updateTask(
                 taskId: taskId,
                 updates: ["isCompleted": isCompleted]
             )
         }
     }
     ```

  4. UI 顯示時區分實例狀態
     ```swift
     ForEach(task.instances ?? []) { instance in
         HStack {
             Text(instance.date, style: .date)
             Spacer()
             Toggle("", isOn: .constant(instance.isCompleted))
                 .onChange(of: instance.isCompleted) { _, newValue in
                     Task {
                         await viewModel.toggleTaskCompletion(
                             taskId: task.id,
                             instanceId: instance.id,
                             isCompleted: newValue
                         )
                     }
                 }
         }
     }
     ```

驗證方法:
   完成一個日期的任務不影響其他日期
   Firestore 中每個實例有獨立的 isCompleted 狀態
   日曆視圖正確顯示各日期的完成狀態

  - studyAssistant/ViewModels/TodoViewModel.swift

問題 1.3.3: Chat 新增任務無法顯示
────────────────────────────────
發現日期: 2025-10-26
嚴重程度:  中等
影響範圍: 聊天助手任務整合

問題描述:
  透過聊天助手新增的任務無法立即在 TodoView 中顯示，
  需要手動重新整理或切換頁面。

根本原因:
  - addTask() 函數使用通知機制觸發重新載入
  - isLoadingTasks 狀態存在競爭條件
  - 通知發送與接收之間有時序問題

解決方案:
  1. 修改 addTask() 函數，直接調用 forceReloadTasks()
     ```swift
     //  舊的實作 (使用通知)
     func addTask(_ task: TodoTask) async {
         // ... 新增任務邏輯
         NotificationCenter.default.post(name: .taskDidUpdate, object: nil)
     }

     //  新的實作 (直接調用)
     func addTask(_ task: TodoTask) async {
         isLoadingTasks = true
     ```

     ```swift
         if let lastTime = lastRequestTime {
             let elapsed = Date().timeIntervalSince(lastTime)
             if elapsed < minimumRequestInterval {
                 let waitTime = minimumRequestInterval - elapsed
                 try? await Task.sleep(nanoseconds: UInt64(waitTime * 1_000_000_000))
             }

         lastRequestTime = Date()
         // ...
     }
     ```

     ```swift
     func sendMessage(_ message: String, retryCount: Int = 0) async {
         do {
             // 發送請求...
         } catch {
             if case let apiError as APIError = error,
                apiError.statusCode == 429,
                retryCount < maxRetries {

                 let waitTime = pow(2.0, Double(retryCount))
                 try? await Task.sleep(nanoseconds: UInt64(waitTime * 1_000_000_000))
     ```

限制 3.1: Firebase 免費方案限制
──────────────────────────────
描述:
  - Firestore 讀取: 50,000 次/天
  - Firestore 寫入: 20,000 次/天
  - Cloud Functions 調用: 125,000 次/月

  - 大量使用者時可能超過配額
  - 需監控用量並優化

解決方案:
  - 實作本地快取減少 Firestore 讀取
  - 批次操作減少寫入次數
  - 考慮升級到付費方案


限制 3.2: OpenAI API 速率限制
────────────────────────────
描述:
  - 免費帳號: 3 RPM (Requests Per Minute)
  - Tier 1: 500 RPM
  - Token 限制依模型而異

影響:
  - 聊天功能可能受限
  - 多使用者同時使用會觸發限制

解決方案:
  - 實作請求隊列
  - 添加冷卻時間
  - 考慮升級 API 方案


限制 3.3: iOS 最低支援版本
─────────────────────────
描述:
  - 目前最低支援 iOS 17.0
  - 部分功能使用 iOS 26+ API

影響:
  - 舊版 iPhone 無法使用

解決方案:
  - 提供替代實作給舊版 iOS

限制 3.4: 重複任務實例數量限制
─────────────────────────────

  - 每個重複任務最多生成 365 個實例

                      五、常見開發陷阱
================================================================================

陷阱 5.1: SwiftUI 在 View 中直接呼叫 async 函數
────────────────────────────────────────────
錯誤範例:
  ```swift
  Button("載入") {
      await viewModel.fetchTasks()  //  錯誤
  }
  ```

正確做法:
  ```swift
  Button("載入") {
      Task {
          await viewModel.fetchTasks()  //  正確
      }
  }
  ```

原因:
  - Button action 不是 async context
  - 需要用 Task {} 包裹


陷阱 5.2: 忘記在 @MainActor 區更新 @Published 屬性
──────────────────────────────────────────────
錯誤範例:
  ```swift
  func fetchData() async {
      let data = try await api.fetch()
      self.items = data  //  可能在背景執行緒
  }
  ```

正確做法:
  ```swift
  @MainActor
  class ViewModel: ObservableObject {
      @Published var items: [Item] = []

      func fetchData() async {
          let data = try await api.fetch()
          self.items = data  //  保證在主執行緒
      }
  }
  ```


陷阱 5.3: Firestore Timestamp 與 Date 轉換
─────────────────────────────────────────
錯誤範例:
  ```swift
  let timestamp = task.startDate  //  類型不明確
  ```

正確做法:
  ```swift
  // Firestore → Swift
  let date = timestamp.dateValue()

  // Swift → Firestore
  let timestamp = Timestamp(date: date)
  ```


陷阱 5.4: 循環參照導致記憶體洩漏
──────────────────────────────
錯誤範例:
  ```swift
  timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { _ in
      self.updateTime()  //  強參照 self
  }
  ```

正確做法:
  ```swift
  timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] _ in
      self?.updateTime()  //  弱參照
  }
  ```


陷阱 5.5: 未取消 Combine 訂閱
────────────────────────────
錯誤範例:
  ```swift
  viewModel.$tasks
      .sink { tasks in
          // 處理任務
      }  //  沒有儲存 AnyCancellable
  ```

正確做法:
  ```swift
  private var cancellables = Set<AnyCancellable>()

  viewModel.$tasks
      .sink { tasks in
          // 處理任務
      }
      .store(in: &cancellables)  //  儲存訂閱
  ```
