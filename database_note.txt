資料庫結構設計
一、資料模型
1. 使用者資料 (User)
- id: String (主鍵)
- username: String (使用者名稱)
- motivationalQuote: String (鼓勵語句)
- targetDate: Date (目標日期)
- userGoal: String (用戶設定的目標)
- createdAt: Date (帳戶創建時間)

2. 待辦事項 (TodoTask)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- title: String (任務標題)
- note: String (任務備註)
- startDate: Date (開始時間)
- endDate: Date (結束時間，若有設定)
- isAllDay: Boolean (是否全天事項)
- colorHex: String (顏色代碼)
- isCompleted: Boolean (是否已完成)
- category: String (分類)
- repeatOption: String (重複選項: 不重複/每天/每週/每月)
- createdAt: Date (建立時間)

3. 計時記錄 (TimerRecord)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- subject: String (學習科目)
- startTime: Date (開始時間)
- endTime: Date (結束時間)
- duration: Int (持續時間，以秒為單位)
- isCompleted: Boolean (是否完成)
- createdAt: Date (建立時間)

4. 聊天記錄 (ChatMessage)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- roomId: String (聊天室ID)
- text: String (訊息內容)
- isMe: Boolean (是否為使用者發送的訊息)
- timestamp: Date (發送時間)

5. 聊天室 (ChatRoom)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- name: String (聊天室名稱)
- createdAt: Date (建立時間)

6. 學習統計 (LearningStatistic)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- subject: String (學科名稱)
- progress: Double (進度百分比，0-1之間)
- totalFocusTime: Int (總專注時間，以秒為單位)
- focusCount: Int (專注次數)
- updatedAt: Date (更新時間)


二、資料庫實作建議
根據您的需求和應用程式設計，我推薦以下方案：
方案一：使用 Core Data (本地儲存)
對於單機應用程式，Core Data 是 Apple 生態系統中最適合的選擇：
優點：
與 SwiftUI 完美整合
無需網絡連接
提供資料持久化、查詢、排序等功能
支援資料模型關係管理
實作步驟：
在 Xcode 中建立 Core Data Model
定義實體和關係
使用 @Environment(\.managedObjectContext) 在 SwiftUI 中管理資料
利用 FetchRequest 獲取資料
方案二：使用 Firebase (雲端儲存)
如果您希望用戶能夠在多設備間同步資料，或考慮未來擴充功能：
優點：
跨設備資料同步
使用者認證功能
實時資料庫
擴展性佳
實作步驟：
設置 Firebase 專案並添加至您的 Swift 專案
使用 Firebase Authentication 管理使用者
使用 Firestore 儲存結構化資料
實作資料同步功能
三、資料關係
以下是資料實體間的關係設計：
一對多關係：
一個使用者可以有多個待辦事項
一個使用者可以有多個計時記錄
一個使用者可以有多個聊天室
一個聊天室可以有多個聊天記錄
一個使用者可以有多個學習統計(不同科目)
查詢考量：
待辦事項可按日期、完成狀態查詢
計時記錄可按科目、日期範圍查詢
聊天記錄可按聊天室ID查詢
四、資料遷移與備份建議
資料遷移計劃：
設計版本控制機制，支援資料模型更新
使用 Core Data 的輕量級遷移功能
備份與還原功能：
實作資料匯出/匯入功能
可考慮使用 iCloud 備份
五、安全考量
本地儲存安全：
敏感資料加密儲存
使用 Keychain 存儲認證信息
網絡傳輸安全：
使用 HTTPS 傳輸
若使用 Firebase，配置適當的安全規則
以上資料庫結構設計基於您的應用程式當前功能，提供了良好的可擴展性，同時考量了各種資料關係和查詢需求。

混合式資料儲存架構分析
優點
離線使用體驗佳：即使沒有網路，使用者仍可使用完整功能
減少網路依賴：只有需要同步的資料才進行網路傳輸
節省雲端資源：相比全部雲端化可降低成本
資料備份機制：本地和雲端雙重保障，降低資料遺失風險
缺點
增加開發複雜度：需管理兩套資料系統及其同步邏輯
可能產生資料衝突：多設備同步時需處理資料合併問題
狀態管理困難：需追蹤資料同步狀態（已同步/待同步）

實現方案
方案：Core Data + Firebase 混合架構
這是最適合您需求的方案，具體實現步驟：

Core Data 作為主要資料庫
儲存所有應用資料
提供離線存取能力
處理大部分資料查詢需求

Firebase 作為同步層
只同步關鍵資料（如待辦事項、學習進度）
處理多設備間資料同步

同步邏輯設計
使用資料時間戳標記資料更新時間
實現資料差異識別機制
設計資料合併策略處理衝突

架構圖
[UI層] <---> [Core Data] <---> [同步服務] <---> [Firebase]
              (本地)            (中間層)          (雲端)

資料選擇建議

適合同步的資料
待辦事項 (TodoTask) - 核心功能，需跨裝置同步
學習統計 (LearningStatistic) - 顯示學習進度
用戶設定 (User) - 個人資料、目標日期

建議本地儲存的資料
聊天記錄 (ChatMessage) - 量大，可選擇性同步
計時詳細記錄 (TimerRecord) - 僅同步摘要數據

資料同步策略
自動同步觸發點：
應用程式啟動時
網路連接恢復時
資料變更後
應用程式進入背景前
衝突解決策略：
使用時間戳決定哪個版本較新
設計合併策略（例如合併不同設備上的待辦清單）
提供用戶手動解決機制（需要時）

專業建議小Tips

時間都用 Timestamp，方便直接在 Firestore Query 比較。
userId全部靠父層Document Id，不用每一筆資料再存一次（省空間⚡）
資料庫安全規則要記得設定，限制只能自己存取自己的資料。
針對大型Collection（像messages），可以用**分頁載入(pagination)**減少一次讀取量！
