資料庫結構設計
一、資料模型
1. 使用者資料 (User) - 需要同步
- id: String (主鍵)
- email: String (使用者信箱，用於認證)
- authProvider: String (認證提供者：Google)
- username: String (使用者名稱)
- motivationalQuote: String (鼓勵語句)
- targetDate: Timestamp (目標日期)
- userGoal: String (用戶設定的目標)
- createdAt: Timestamp (帳戶創建時間)
- isActive: Boolean (帳號是否啟用)
- lastLoginAt: Timestamp (最後登入時間)
- vip: Boolean (是否為vip)
- isDeleted: Boolean (軟刪除標記)

2. 待辦事項 (TodoTask) - 需要同步
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- title: String (任務標題)
- note: String (任務備註)
- notificationId: String?
- startDate: Timestamp (開始時間)
- endDate: Timestamp? (結束時間，若有設定)
- isAllDay: Boolean (是否全天事項)
- colorHex: String (顏色代碼)
- isCompleted: Boolean (是否已完成)
- category: String (分類)
- repeatOption: String (重複選項: 不重複/每天/每週/月)
- createdAt: Timestamp (建立時間)
- lastModifiedAt: Timestamp (最後修改時間)
- lastSyncedAt: Timestamp? (最後同步時間)
- version: Int (數據版本，用於解決衝突)
- syncStatus: String (同步狀態: 待同步/已同步/衝突)
- isDeleted: Boolean (軟刪除標記)

3. 計時記錄 (TimerRecord) - 僅本地儲存 (僅同步摘要數據)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- subject: String (學習科目)
- startTime: Timestamp (開始時間)
- endTime: Timestamp (結束時間)
- duration: Int (持續時間，以秒為單位)
- isCompleted: Boolean (是否完成)
- createdAt: Timestamp (建立時間)
- lastModifiedAt: Timestamp (最後修改時間)
- isDeleted: Boolean (軟刪除標記) // 保留但可能不使用

4. 聊天記錄 (ChatMessage) - 僅本地儲存 (選擇性同步)
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- roomId: String (聊天室ID)
- text: String (訊息內容)
- isMe: Boolean (是否為使用者發送的訊息)
- timestamp: Timestamp (發送時間)
- isDeleted: Boolean (軟刪除標記) // 保留但可能不使用

5. 聊天室 (ChatRoom) - 需要同步
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- name: String (聊天室名稱)
- createdAt: Timestamp (建立時間)
- lastModifiedAt: Timestamp (最後修改時間)
- isDeleted: Boolean

5. 助理個人化設定 - 需要同步
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- lastModifiedAt: Timestamp (最後修改時間)
- isDeleted: Boolean

一週七天的可用時間
struct StudySchedulePreference: Codable, Identifiable {
    var id: String // 通常與 userId 一樣
    var userId: String
    var weeklyAvailableSlots: [WeekdaySlot] // 一週七天的可用時間
    var updatedAt: Date
}

struct WeekdaySlot: Codable {
    var weekday: Int    // 1 = Sunday, 7 = Saturday（或你自己定義）
    var timeSlots: [TimeRange] // 每天可以讀書的時段
}

struct TimeRange: Codable {
    var startHour: Int // 0~23
    var endHour: Int   // 0~23
}


6. 學習統計 (LearningStatistic) - 需要同步
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- subject: String （類別）
- progress: Double (進度百分比，0-1之間)
- totalFocusTime: Int (總專注時間，以分鐘為單位)
- updatedAt: Timestamp (更新時間)
- version: Int (數據版本)

7. 通知/提醒 (Notification) - 需要同步
- id: String (主鍵)
- userId: String (外鍵，關聯使用者)
- title: String (通知標題)
- body: String (通知內容)
- type: String (通知類型：任務提醒/學習提醒/系統通知)
- relatedItemId: String? (關聯項目ID，如任務ID或計時記錄ID)
- relatedItemType: String? (關聯項目類型：待辦事項/計時/聊天等)
- scheduledFor: Timestamp (計劃通知時間)
- repeatType: String? (重複類型：無/每日/每週/每月/自定義)
- isRead: Boolean (是否已讀)
- isDelivered: Boolean (是否已發送)
- isActive: Boolean (是否啟用)
- createdAt: Timestamp (建立時間)
- lastModifiedAt: Timestamp (最後修改時間)
- isDeleted: Boolean (軟刪除標記)

二、資料庫實作建議
根據您的需求和應用程式設計，我推薦以下方案：
方案一：使用 Core Data (本地儲存)
對於單機應用程式，Core Data 是 Apple 生態系統中最適合的選擇：
優點：
與 SwiftUI 完美整合
無需網絡連接
提供資料持久化、查詢、排序等功能
支援資料模型關係管理
實作步驟：
在 Xcode 中建立 Core Data Model
定義實體和關係
使用 @Environment(\.managedObjectContext) 在 SwiftUI 中管理資料
利用 FetchRequest 獲取資料

方案二：使用 Firebase (雲端儲存)
如果您希望用戶能夠在多設備間同步資料，或考慮未來擴充功能：
優點：
跨設備資料同步
使用者認證功能
實時資料庫
擴展性佳
實作步驟：
設置 Firebase 專案並添加至您的 Swift 專案
使用 Firebase Authentication 管理使用者
使用 Firestore 儲存結構化資料

三、資料關係
以下是資料實體間的關係設計：
一對多關係：
一個使用者可以有多個待辦事項
一個使用者可以有多個計時記錄
一個使用者可以有多個聊天室
一個聊天室可以有多個聊天記錄
一個使用者可以有多個學習統計(不同科目)
查詢考量：
待辦事項可按日期、完成狀態查詢
計時記錄可按科目、日期範圍查詢
聊天記錄可按聊天室ID查詢
四、資料遷移與備份建議
資料遷移計劃：
設計版本控制機制，支援資料模型更新
使用 Core Data 的輕量級遷移功能
備份與還原功能：
實作資料匯出/匯入功能
可考慮使用 iCloud 備份
五、安全考量
本地儲存安全：
敏感資料加密儲存
使用 Keychain 存儲認證信息
網絡傳輸安全：
使用 HTTPS 傳輸
若使用 Firebase，配置適當的安全規則
以上資料庫結構設計基於您的應用程式當前功能，提供了良好的可擴展性，同時考量了各種資料關係和查詢需求。

混合式資料儲存架構分析
優點
離線使用體驗佳：即使沒有網路，使用者仍可使用完整功能
減少網路依賴：只有需要同步的資料才進行網路傳輸
節省雲端資源：相比全部雲端化可降低成本
資料備份機制：本地和雲端雙重保障，降低資料遺失風險
缺點
增加開發複雜度：需管理兩套資料系統及其同步邏輯
可能產生資料衝突：多設備同步時需處理資料合併問題
狀態管理困難：需追蹤資料同步狀態（已同步/待同步）

專業建議小Tips

時間都用 Timestamp，方便直接在 Firestore Query 比較。
userId全部靠父層Document Id，不用每一筆資料再存一次（省空間⚡）
資料庫安全規則要記得設定，限制只能自己存取自己的資料。
針對大型Collection（像messages），可以用**分頁載入(pagination)**

常見陷阱
不要在 View 裡直接呼叫 async 函數 —— 要包在 Task {} 裡。
如果使用 ObservableObject + @Published 管理資料，請在 @MainActor 區更新資料以避免錯誤。
若你要用 .addDocument()，要記得資料符合 Codable。

主要資料模型
1. 用戶與設定相關
UserProfile: 使用者個人資料
屬性：id, email, username, motivationalQuote, targetDate, userGoal, learningStage, isVIP, lastLoginAt, version, lastSyncedAt, isDeleted
功能：儲存使用者個人資訊和目標
AppSettings: 應用程式設定
屬性：id, isDarkMode, notificationsEnabled, isShockEnabled, lastModified
功能：儲存使用者對應用的偏好設定
2. 任務管理相關
TodoTask: 待辦任務
屬性：id, title, note, color, focusTime, category, isAllDay, isCompleted, repeatType, startDate, endDate, createdAt, userId
功能：表示一個待辦事項任務，支援重複任務
RepeatType: 任務重複類型（枚舉）
選項：none, daily, weekly([Int]), monthly([Int])
功能：定義任務的重複方式，支援每日、每週和每月重複
3. 計時器相關
TimerRecord: 計時記錄
屬性：id, userId, subject, startTime, endTime, duration, isCompleted, createdAt, lastModifiedAt, isDeleted
功能：記錄使用者的學習時間和科目
TimerStatistics: 計時統計資料
屬性：subjectStats, dailyStats, totalDuration, totalSessions, completedSessions, incompleteSessions, totalTime, averageSessionTime
功能：匯總計時器記錄的統計資訊
SubjectStats: 科目統計
屬性：totalDuration, sessionCount, completedCount
功能：按科目統計計時資訊
DailyStats: 日期統計
屬性：totalDuration, sessionCount, subjects
功能：按日期統計計時資訊
4. 服務相關
DataServiceError: 資料服務錯誤（枚舉）
選項：networkError, syncError, validationError, notFound, permissionDenied, unknown
功能：定義資料服務可能出現的錯誤類型
SyncStatus: 同步狀態（枚舉）
選項：notSynced, syncing, synced, error
功能：表示資料同步狀態
視圖模型 (ViewModels)
TodoViewModel: 待辦任務視圖模型
功能：管理待辦任務的 CRUD 操作和資料同步
UserSettingsViewModel: 使用者設定視圖模型
功能：管理使用者設定、個人資料的讀取和更新
TimerManager: 計時器管理器
功能：管理計時器的運行、暫停、重置和記錄
ProfileViewModel: 個人資料視圖模型
功能：管理使用者個人資料的讀取和更新
服務介面
DataServiceProtocol: 資料服務協議
功能：定義資料操作的共用介面
FirebaseService: Firebase 資料服務
功能：實現資料服務協議，處理與 Firebase 的資料交互
這些是專案中的主要資料結構，它們共同構成了應用程式的資料層和業務邏輯層。已成功移除了 TimerSettings 相關的程式碼，並保持了其他功能的完整性。
