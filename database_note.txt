1. 資料庫架構概覽
===============================================================================

本專案使用 Firebase Firestore 作為主要資料庫，採用分層結構設計：

架構特點：
  - 使用 Cloud Functions 作為後端邏輯層
  - 客戶端透過 Cloud Functions 間接操作 Firestore
  - 支援離線快取和樂觀更新
  - 使用文檔 ID 作為主鍵

2. Collection 結構詳解
===============================================================================

2.1 tasks 集合
-------------------------------------------------------------------------------

路徑結構：
  /tasks/{userId}/userTasks/{taskId}
  /tasks/{userId}/userTasks/{taskId}/instances/{instanceId}

說明：
  - 任務資料以 userId 分層，每個使用者有獨立的 userTasks 子集合
  - 重複任務會在 instances 子集合中生成多個實例
  - 每個實例代表重複任務在特定日期的一次出現

2.1.1 userTasks 文檔結構
-------------------------------------------------------------------------------
欄位名稱              | 類型           | 說明
-------------------------------------------------------------------------------
title                 | String         | 任務標題
note                  | String         | 任務備註
color                 | Object         | 顏色資料 {red, green, blue, opacity}
focusTime             | Number         | 專注時間（分鐘）
category              | String         | 任務類別（學習、工作等）
isAllDay              | Boolean        | 是否全天任務
isCompleted           | Boolean        | 是否完成（非重複任務）
repeatType            | Object         | 重複類型 {type: "none"/"daily"/"weekly"/"monthly"}
startDate             | Timestamp      | 任務開始日期時間
endDate               | Timestamp      | 任務結束日期時間
repeatEndDate         | Timestamp?     | 重複結束日期（重複任務專用）
createdAt             | Timestamp      | 建立時間
userId                | String         | 使用者 ID
notificationEnabled   | Boolean        | 是否啟用通知
notificationOffset    | Number         | 提前提醒時間（分鐘）
-------------------------------------------------------------------------------

2.1.2 instances 文檔結構
-------------------------------------------------------------------------------
欄位名稱              | 類型           | 說明
-------------------------------------------------------------------------------
date                  | Timestamp      | 實例日期
isCompleted           | Boolean        | 此實例是否完成
parentTaskId          | String         | 父任務 ID
createdAt             | Timestamp      | 建立時間
-------------------------------------------------------------------------------

實例生成規則：
  - 每天重複：生成 365 個實例（一年份）
  - 每週重複：生成 52 個實例
  - 每月重複：生成 12 個實例
  - 超過 repeatEndDate 的實例不生成

3.1 TodoTask (任務模型)
-------------------------------------------------------------------------------

檔案: studyAssistant/Models/TodoTask.swift

struct TodoTask: Identifiable, Codable, Equatable {
    let id: String                      // 任務 ID (新任務=UUID, 從 Firestore=文檔ID)
    var title: String                   // 任務標題
    var note: String                    // 任務備註
    var color: Color                    // 任務顏色
    var focusTime: Int                  // 專注時間（分鐘）
    var category: String                // 任務類別
    var isAllDay: Bool                  // 是否全天
    var isCompleted: Bool               // 是否完成
    var repeatType: RepeatType          // 重複類型
    var startDate: Date                 // 開始日期
    var endDate: Date                   // 結束日期
    var repeatEndDate: Date?            // 重複結束日期
    var createdAt: Date                 // 建立時間
    var userId: String                  // 使用者 ID
    var instances: [TaskInstance]       // 任務實例陣列
    var notificationEnabled: Bool       // 是否啟用通知
    var notificationOffset: Int         // 提前提醒時間（分鐘）
}

ID 生成規則：
  - 新建任務：使用 UUID().uuidString（36 字符）
  - 從 Firestore 載入：使用 Firestore document ID（~20 字符）
  - 更新判斷：透過 ID 長度區分（<36 為現有任務，調用 updateTask）

RepeatType 枚舉：
  - .none: 不重複
  - .daily: 每天重複
  - .weekly: 每週重複
  - .monthly: 每月重複

4. Cloud Functions API
===============================================================================

所有 Cloud Functions 部署在 asia-east1 區域

4.1 任務管理 API
-------------------------------------------------------------------------------

createTask
-------------------------------------------------------------------------------
功能：創建新任務（包含自動生成實例）
參數：{ task: TodoTask }
回傳：{ success: Boolean, taskId?: String, error?: String }
說明：
  - 自動生成重複任務實例
  - 若無 repeatEndDate，預設為一年後
-------------------------------------------------------------------------------

updateTask
-------------------------------------------------------------------------------
功能：更新現有任務
參數：{ taskId: String, updates: Partial<TodoTask> }
回傳：{ success: Boolean, error?: String }
說明：
  - 檢查是否需要重新生成實例
  - 保留已完成的實例，重新生成未完成的實例
-------------------------------------------------------------------------------

deleteTask
-------------------------------------------------------------------------------
功能：刪除任務（包含所有實例）
參數：{ taskId: String }
回傳：{ success: Boolean, error?: String }
-------------------------------------------------------------------------------

fetchTasks
-------------------------------------------------------------------------------
功能：獲取所有任務（包含實例）
參數：無
回傳：{ success: Boolean, tasks: TodoTask[], error?: String }
說明：
  - 按 createdAt 降序排列
  - 自動載入所有實例
-------------------------------------------------------------------------------

toggleTaskCompletion
-------------------------------------------------------------------------------
功能：切換非重複任務的完成狀態
參數：{ taskId: String, isCompleted: Boolean }
回傳：{ success: Boolean, error?: String }
-------------------------------------------------------------------------------

5. 資料流向說明
===============================================================================

5.1 任務建立流程
-------------------------------------------------------------------------------

1. 使用者在 TodoAddView 輸入任務資料
2. 點擊「儲存」→ TodoViewModel.saveNewTask()
3. 呼叫 FirebaseService.saveTodoTask()
4. 判斷 task.id 長度：
   - ID 長度 = 36（UUID）→ 新任務 → 呼叫 createTask Cloud Function
   - ID 長度 < 36（Firestore ID）→ 更新任務 → 呼叫 updateTask Cloud Function
5. Cloud Function 執行：
   - 儲存任務資料到 /tasks/{userId}/userTasks/{taskId}
   - 若為重複任務，自動生成實例到 instances 子集合
6. 排程通知（若啟用）
7. 重新載入任務列表


5.2 任務更新流程
-------------------------------------------------------------------------------

1. 使用者在 TodoEditView 修改任務
2. 點擊「儲存」→ TodoViewModel.updateTask()
3. 呼叫 FirebaseService.saveTodoTask()
4. 根據 ID 長度判���：呼叫 updateTask Cloud Function
5. Cloud Function 執行：
   - 比對任務資料，判斷是否需要重新生成實例
   - 若時間或重複設定變更，保留已完成實例，重新生成未完成實例
   - 更新任務資料
6. 更新通知
7. 更新本地任務列表


5.3 任務完成流程
-------------------------------------------------------------------------------

非重複任務：
1. 點擊完成按鈕 → TodoViewModel.toggleTaskCompletion()
2. 立即更新本地狀態（樂觀更新）
3. 加入更新佇列 → processCompletionQueue()
4. 呼叫 toggleTaskCompletion Cloud Function
5. 若失敗且重試次數 < 3，重新加入佇列
6. 超過重試次數，恢復本地狀態並提示錯誤

重複任務實例：
1. 點擊實例完成按鈕 → TodoViewModel.toggleInstanceCompletion()
2. 立即更新本地狀態
3. 呼叫 updateTaskInstance Cloud Function
4. 若失敗，恢復本地狀態


5.4 通知流程
-------------------------------------------------------------------------------

啟動時：
1. App 啟動 → studyAssistantApp.init()
2. 同步全域通知設定到 NotificationManager.shared
3. NotificationManager.globalNotificationOffsetMinutes = 使用者設定值

任務建立/更新時：
1. 檢查 task.notificationEnabled
2. 若啟用，呼叫 NotificationManager.scheduleNotification(for: task)
3. 使用全域偏移時間（globalNotificationOffsetMinutes）
4. 重複任務：根據類型預先排程有限數量的通知
   - 每天：7 天
   - 每週：4 週
   - 每月：3 個月

通知格式：
  標題：任務名稱 + 時間範圍
    範例：「複習微積分  14:00 - 16:00」
  內容：任務備註（若為空則為單一空格）
    範例：「複習第三章極限與連續性」


5.5 離線支援
-------------------------------------------------------------------------------

快取機制：
- 任務列表快取時間：60 秒
- 快取檢查：loadTasks() 會檢查 lastTasksLoadTime
- 強制重新載入：forceReloadTasks() 忽略快取

樂觀更新：
- 完成狀態切換：立即更新本地，背景同步
- 使用更新佇列：避免頻繁的網路請求
- 失敗處理：重試機制 + 錯誤恢復


===============================================================================
附錄：重要技術決策
===============================================================================

1. 為什麼使用 Cloud Functions？
   - 集中管理業務邏輯
   - 統一處理重複任務實例生成
   - 防止客戶端直接操作資料庫導致的資料不一致

2. 為什麼任務實例採用預先生成？
   - iOS 通知數量限制（64 個）
   - 避免每次查詢時動態計算（效能考量）
   - 支援離線場景

3. 為什麼通知使用全域設定而非個別任務設定？
   - 簡化使用者體驗
   - 避免重複設定
   - 減少資料儲存量
   - 統一管理方便同步

4. 為什麼 TimerRecord 使用 UserDefaults 而非 Firestore？
   - 計時資料為即時性資料，不需雲端同步
   - 減少 Firestore 讀寫次數
   - 降低成本
   - 提升效能

5. 任務 ID 長度判斷邏輯
   - UUID: 36 字符（含連字符）
   - Firestore ID: 20 字符
